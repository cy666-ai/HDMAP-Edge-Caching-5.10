================================================================================
HLR-Cache 边缘缓存系统 — 缓存策略详解
================================================================================

## 核心概念
- RSU (Road-Side Unit)：路边单元，部署在道路沿线的缓存节点
- Tile (瓦片)：地图数据的基本单位，每个 RSU 最多缓存 100 个瓦片 (CHUNKS_PER_RSU = 100)
- Hit Rate (命中率)：车辆请求的瓦片在 RSU 缓存中的命中比例
- Coverage (覆盖范围)：RSU 通信半径 300 米 (RSU_PROXIMITY_M = 300)

## MWC 算法 (Maximum Weight Closure)
最大权重闭包算法是本系统的核心缓存优化算法：
- 将有向无环图中的缓存依赖关系建模为图的最大权重闭包问题
- 通过 networkx 的最小割/最大流算法求解
- 目标：在 RSU 容量约束下最大化期望命中率

## 缓存决策流程
1. 概率分布建模：基于 Poisson 分布 + Power-law 分布计算每块瓦片的请求概率
2. 依赖图构建：分析瓦片之间的引用依赖关系
3. 效用计算：综合请求概率、瓦片大小、传输成本计算净效用 W_net
4. MWC 求解：将效用和依赖关系输入 networkx min-cut 求解
5. 容量优化：每 RSU 按 Capacity_Scale=2.0 分配缓存空间
6. 依赖修复：确保关键依赖瓦片不被遗漏

## 关键参数
- CHUNKS_PER_RSU = 100：每个 RSU 可缓存的瓦片数量
- RSU_PROXIMITY_M = 300：车辆与 RSU 的通信距离（米）
- ALGO_INTERVAL_TICKS = 5：每 5 个模拟 tick 执行一次 MWC 算法
- alpha = 0.055：净效用计算中的成本惩罚系数
- Capacity_Scale = 2.0：RSU 容量倍增因子
- BATCH_SIZE = 3：概率分布每 3 个 RSU 重置一次，避免概率扩散

## 命中率计算
命中率 = 命中瓦片数 / 总请求瓦片数
- 系统级命中率 CHR_Total：所有路线的加权平均
- 路线级命中率 CHR_RSU[n]：每条路线各 RSU 的独立命中率
- 只统计车辆在 RSU 覆盖范围内发起的请求

## RSU 部署策略
- RSU 沿路线按 500 米间距部署（Haversine 公式计算）
- 跨路线 RSU 去重：250 米半径内视为同一 RSU
- 每条路线通常有 8-14 个 RSU（取决于路线长度）
- 系统总 RSU 数量约 68 个

## 对外接口
- 输入：_vehicle_input.json（算法参数 + 路线数据）
- 输出：cache_decision.json（每路线 CacheDecision 布尔数组 + psi + W_net + CHR）
- Node.js 通过 child_process.spawn('python') 调用，以 JSON 文件交换数据
